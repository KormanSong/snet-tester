# SNET Protocol Tester UI 재배치 MRS

## Goal
- 현재 `SNET Protocol Tester` UI를 산업 제어용 운영 화면 관점에서 재정리해, 상태 가시성, 오조작 방지, 채널 중심 작업 흐름을 강화한다.

## 현재 화면 목적 추정
- 프로토콜 기반 채널 제어값을 설정한다.
- 실시간 응답과 추이를 관찰한다.
- 채널별 상태와 보정값을 확인한다.
- PID 및 프레임 디버그를 수행한다.

## 명시 요구
- 메인 화면은 `대시보드 + 우측 작업 영역` 구조를 우선한다.
- 밝은 회색 기반의 클래식 산업용 UI 톤을 유지한다.
- 정보 밀도는 높게 가져가되, 일반 작업자도 읽기 쉬워야 한다.
- 로그와 알람은 작업 중에도 고정 노출한다.
- 메인 창은 가능하면 단일 `.ui` 또는 단일 메인 화면 구조로 관리한다.
- 신뢰감, 전문성, 정돈됨을 우선한다.

## 숨은 요구 가설
- 운영자에게 가장 중요한 것은 기능 수보다 `지금 장비가 어떤 상태인지`를 즉시 파악하는 것이다.
- `RUN`, `STOP`, `CAL`, `SET`는 같은 화면에 있더라도 의미상 다른 계층으로 분리되어야 한다.
- `--`, `0`, 빈 칸은 모두 다른 의미일 가능성이 높아 명시적으로 구분되어야 한다.
- 채널 선택, 현재값 확인, 목표값 적용, 결과 확인이 한 흐름으로 이어져야 한다.
- Calibration, PID, Frame Debug는 운영 화면에 남기되, 기본 운영 흐름보다 한 단계 아래로 내려야 한다.

## 사용자 작업 시나리오
1. 작업자가 연결 상태와 현재 모드를 확인한다.
2. 선택 채널 또는 프리셋을 고른다.
3. 목표 비율 또는 제어값을 입력한다.
4. 적용 전 위험 동작 여부를 확인한다.
5. 운전을 시작하거나 중지한다.
6. 좌측 추세와 채널 요약에서 응답을 확인한다.
7. 이상 발생 시 하단 알람/이벤트 로그에서 원인과 조치 힌트를 본다.
8. 필요할 때만 보정 또는 엔지니어링 탭으로 진입한다.

## 작업 목록
1. 상태 체계를 재정의한다.
   - 운전 상태, 연결 상태, 모드, 알람 상태를 단일 상단 상태 바에 통합
   - Priority: P0
2. 로그/알람 구조를 고정 노출 방식으로 정리한다.
   - 이벤트 시간, 심각도, 메시지, ACK 여부, 권장 조치 표시
   - Priority: P0
3. 감시와 제어를 분리한 메인 레이아웃을 설계한다.
   - 좌측 감시, 우측 제어/설정, 하단 알람
   - Priority: P0
4. 채널 중심 흐름으로 작업 패널을 재구성한다.
   - 선택 채널, 현재값, 목표값, 적용 버튼, 결과 피드백 연결
   - Priority: P1
5. 위험 동작 보호 장치를 정의한다.
   - STOP, CAL, 저장, 삭제, 프리셋 적용 확인 절차
   - Priority: P1
6. 용어와 상태 표기를 표준화한다.
   - `--`, `N/A`, `Disconnected`, `Disabled`, `No Response` 구분
   - Priority: P1
7. 엔지니어링 영역을 보조 패널 또는 탭 구조로 격리한다.
   - PID, Calibration, TX/RX Frame
   - Priority: P2

## Open Questions
- 1차 사용자는 현장 작업자인가, 시험 엔지니어인가, 유지보수 담당자인가?
- `RUN`, `STOP`, `CAL`, `SET` 중 확인 절차가 반드시 필요한 위험 동작은 무엇인가?
- 별도 알람/로그 화면이 이미 존재하는가, 아니면 메인 화면 내 통합이 필요한가?
- 채널은 물리 장치와 1:1 대응인가, 가상 채널/프리셋 개념이 섞여 있는가?
- `PRESS`, `TEMP`, `AD`, `F.에어`의 실제 용어 정의와 단위는 무엇인가?

## Risks
- 상태 의미 불일치 리스크
  - 현재 UI의 `RUN`, `WAIT`, `LINK`, `CAL` 의미가 코드/장비 프로토콜과 다를 수 있다.
- 운영자/엔지니어 혼합 리스크
  - 운영 화면과 디버그 기능을 한 수준으로 노출하면 오조작 위험이 커진다.
- 데이터 의미 해석 리스크
  - `--`와 `0`의 의미를 잘못 정리하면 실제 현장 판단에 오류를 만든다.
- 범위 확장 리스크
  - UI 재배치가 용어 표준화, 로그 정책, 인터락 정책까지 확장될 가능성이 높다.
- 기존 사용자 습관 리스크
  - 지금 화면에 익숙한 사용자는 레이아웃 변경을 불편하게 느낄 수 있다.
- 구현 리스크
  - 기존 `main_window.ui`를 바로 수정하면 기능 회귀가 발생할 수 있어, 별도 프리뷰와 사양 확정이 선행되어야 한다.

## Success Criteria
- 화면 최상단만 봐도 연결, 운전, 모드, 알람 상태를 2초 안에 파악할 수 있다.
- 사용자가 선택 채널과 적용 대상 값을 혼동하지 않는다.
- 위험 동작은 실수로 1회 클릭만으로 실행되지 않는다.
- 알람/이벤트가 화면 전환 없이 상시 확인 가능하다.
- 운영 업무와 엔지니어링 업무가 같은 화면 안에서도 시각적으로 분리된다.
